# Citation Graph Explorer v2.1 — Taxonomy, Enrichment, Subtopic Drilling

**Date:** 2026-04-04
**Status:** Approved
**Depends on:** `2026-04-04-citation-graph-v2-design.md` (implemented on `feature/graph-v2`)
**Goal:** Make the graph dashboard genuinely useful for litigating advocates by fixing issue classification consistency, enriching Neo4j with PostgreSQL metadata, and adding subtopic drilling.

## Problem Statement

1. **Issue classification inconsistency** — The LLM produces variant tags for the same concept: `criminal_law.murder` (119), `criminal.murder` (34), `constitutional_law.article_14` (64), `fundamental_rights.article_14` (41), `constitutional.article_14` (32). Lawyers miss 40% of relevant cases.

2. **Neo4j is starved** — PostgreSQL has 100+ fields per case. Neo4j has 12. The graph can't surface subtopics, statute sections, bench composition, or disposal patterns.

3. **No subtopic drilling** — Clicking "Criminal Law (4,284)" shows a flat list. Lawyers need to narrow to "Bail → Anticipatory Bail" or "Section 302 IPC."

## Three-Layer Solution

### Layer 1: Canonical Legal Taxonomy (Ingestion)

**18 top-level categories, ~150 subtopics** based on the Supreme Court's 48/189 system, distilled to what appears in SC judgments:

| Category | Example Subtopics |
|---|---|
| Criminal Law | Murder, Bail, Anticipatory Bail, Sentencing, NDPS, POCSO, Rape, Dowry, Quashing, Investigation |
| Constitutional Law | Article 14, Article 19, Article 21, Article 32, Article 226, Article 142, PIL, Federalism |
| Civil Procedure | Execution, Revision, Review, Limitation, Order 7 Rule 11, Res Judicata, Injunction |
| Land & Property | Land Acquisition, Compensation, Transfer of Property, Specific Performance, Adverse Possession |
| Tax Law | Income Tax, GST, Customs, Excise |
| Labour & Service | Recruitment, Promotion, Seniority, Disciplinary Proceedings, Termination, Pension |
| Arbitration | Appointment of Arbitrator, Section 34 Challenge, Section 11, Enforcement of Award |
| Family Law | Divorce, Maintenance, Custody, Succession |
| Insolvency | CIRP, Liquidation, Moratorium, Resolution Plan |
| Company Law | Oppression, Mismanagement, Winding Up |
| Contract & Commercial | Cheque Dishonour, Specific Relief, Damages |
| Environmental Law | Pollution, Mining, Forest, Wildlife |
| Evidence | Circumstantial Evidence, Burden of Proof, Dying Declaration |
| Motor Accident & Tort | Compensation, Negligence |
| Consumer Protection | Deficiency of Service, Medical Negligence, Insurance |
| Administrative Law | Judicial Review, Natural Justice, Legitimate Expectation |
| Election Law | Disqualification, Corrupt Practices |
| Regulatory Law | Electricity, Telecom, Competition, Banking & Finance |

**Storage:** Python dict in `backend/app/core/legal/taxonomy.py`. The LLM prompt includes the full taxonomy and instructs: "Classify using ONLY tags from this list."

**Normalization mapper** — post-LLM step mapping variant tags to canonical forms:
```python
NORMALIZATION_MAP = {
    "criminal.murder": "criminal_law.murder",
    "fundamental_rights.article_14": "constitutional_law.article_14",
    "constitutional.article_14": "constitutional_law.article_14",
    "constitutional.article_226": "constitutional_law.article_226",
    "criminal_procedure.bail": "criminal_law.bail",
    # ... ~50 mappings for observed variants
}
```
Runs after LLM extraction, before DB write. Unmapped tags fall through unchanged (no data loss).

**New extraction field: `primary_legal_issue`** — Single natural language sentence: "What is THE central legal question this case decides?" Stored in PostgreSQL + pushed to Neo4j.

**Safety:** All changes are additive. Existing `issue_classification` field type unchanged (`Array[String]`). New field is a new column. No downstream retrieval impact.

### Layer 2: Neo4j Enrichment (Analytics Script)

**New Case node properties (8):**

| Property | Source (PG column) | Lawyer value |
|---|---|---|
| `jurisdiction` | `jurisdiction` | Filter by legal domain |
| `coram_size` | `coram_size` | Larger bench = stronger authority |
| `is_reportable` | `is_reportable` | Reportable = higher precedential value |
| `opinion_type` | `opinion_type` | Unanimous vs split affects persuasiveness |
| `issue_tags` | `issue_classification` (normalized) | Subtopic drilling |
| `primary_legal_issue` | `primary_legal_issue` | One-line case summary |
| `fact_pattern_summary` | `fact_pattern_summary` | Quick distinguishability check |
| `headnote_text` | `headnotes[0].proposition` | Core holding |

**New node types:**

1. **`StatuteSection`** — `{id, act, section, section_title}`
   - Edge: `(Case)-[:INTERPRETS]->(StatuteSection)` from `statute_sections_interpreted`
   - Example: `{id: "ipc_302", act: "Indian Penal Code", section: "Section 302"}`

2. **`IssueTopic`** — `{id, category, subtopic, label}`
   - Edge: `(Case)-[:CLASSIFIED_AS]->(IssueTopic)`
   - Example: `{id: "criminal_law.bail", category: "Criminal Law", subtopic: "Bail"}`

**Implementation:** New functions in `compute_graph_analytics.py`:
- `enrich_neo4j_from_postgres()` — Batch read PG → batch write Neo4j properties
- `create_statute_section_nodes()` — MERGE StatuteSection nodes + INTERPRETS edges
- `create_issue_topic_nodes()` — MERGE IssueTopic nodes + CLASSIFIED_AS edges
- `normalize_issue_tags(tags)` — Apply canonical mapping

### Layer 3: Dashboard Subtopic Drilling (Frontend)

**Subtopic expansion** — Clicking a top-level category expands to show subtopics:
```
Criminal Law (4,284)
  ├─ Murder (119)
  ├─ Bail (60)
  ├─ Sentencing (30)
  └─ + 12 more...
```
Click subtopic → dashboard filters to that subtopic.

**Statute section tab** — Toggle between `[By Topic]` and `[By Statute]`:
```
Section 302 IPC (87)  |  Article 21 COI (152)  |  Section 34 ACA (45)
```
Click → see cases interpreting that section with `interpretation_summary` in cards.

**Enhanced case cards:**
- Bench size badge: "5J", "3J", "2J"
- Primary legal issue (one-line)
- Disposal tag: "Allowed" / "Dismissed" / "Remanded"
- Reportable indicator: "R" badge

**Filter bar** (below topic pills):
```
Bench: [Any ▾]  Disposal: [Any ▾]  Year: [2020 ─── 2026]  Reportable: [☐]
```

### API Changes

**Modified: `GET /graph/dashboard`**
New optional query params: `subtopic`, `statute_section`, `bench_type`, `disposal_nature`, `year_from`, `year_to`, `is_reportable`

Response adds:
```json
{
  "subtopics": [{"tag": "criminal_law.murder", "label": "Murder", "count": 119}],
  "statute_sections": [{"id": "ipc_302", "act": "IPC", "section": "Section 302", "count": 87}]
}
```

**New: `GET /graph/subtopics?category=criminal_law`** — Subtopics for a category with counts. Cached 1hr.

**New: `GET /graph/statute-sections`** — All StatuteSection nodes with case counts. Cached 1hr.

**No changes to existing endpoints.** Neighborhood, chain, path, treatment-summary all continue working. They return new node properties automatically.

## Safety Guarantees

- All ingestion changes are additive (new field, constrained existing field)
- PostgreSQL column types unchanged
- Pinecone vector types and metadata schema unchanged
- Search/retrieval queries unchanged
- Existing Neo4j nodes/edges untouched (new properties added, new node types added)
- Analytics enrichment runs as separate post-ingestion step
- Normalization preserves unmapped tags (no data loss)
