# Phase 6.5: Quality Excellence — Design

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Fix critical quality gaps and add competitive moat features so Smriti delivers lawyer-grade responses that beat Harvey, Jhana, and CaseMine on Indian legal research quality.

**Architecture:** Backend-first quality fixes (RAG grounding, chunk-level retrieval, search strategy), then competitive features (equivalent citations, structured judgment decomposition, precedent strength classification), then frontend UX to make quality visible.

**Tech Stack:** Existing stack — no new dependencies. PostgreSQL (new tables), Pinecone (metadata filters), Gemini (prompt improvements), React (new components).

---

## Motivation

A quality audit revealed that while Smriti's architecture is sound (hybrid search, RRF, Cohere reranking, LangGraph agents), the actual response quality has critical gaps:

1. **RAG context sends only case metadata to the LLM** — no actual judgment text, ratio decidendi, or matching chunks. The model answers legal questions blind.
2. **Search strategy field is computed but never used** — citation lookups run full vector search unnecessarily.
3. **No anti-sycophancy** — AI agrees with incorrect legal premises instead of flagging them.
4. **All precedents treated equally** — no distinction between binding Constitution Bench decisions and persuasive single-judge orders.
5. **No equivalent citation matching** — AIR and SCC citations for the same case are treated as unrelated.
6. **No structured judgment decomposition** — can't search within "holdings only" or "facts only".

Competitors like LegitQuest (iDRAF structured search), Jhana (16M docs, judge-trusted), and Harvey (paragraph-level grounding) already solve some of these. We need to match and exceed them.

## 1. Fix RAG Context Grounding (CRITICAL)

### Problem
`_format_context()` in `backend/app/core/chat/rag.py` sends only title/citation/court/year to the LLM. The `ratio_decidendi` and `description` fields are fetched from PostgreSQL but dropped before reaching the prompt.

### Solution

**1a. Chunk-level retrieval** — Modify `_vector_search()` to return chunk text alongside case_id/score. Pinecone already stores `text` (1000 chars) in metadata. Keep the top matching chunks per case for RAG context.

**1b. Rich context format** — The context sent to the LLM becomes:

```
[1] State of Maharashtra v. XYZ
    Citation: (2023) 5 SCC 123
    Court: Supreme Court of India (Division Bench) | Year: 2023
    Bench: Hon'ble Justice A, Hon'ble Justice B

    Ratio Decidendi:
    The court held that... [up to 2000 chars]

    Relevant Passage:
    "...the principle established in this case is..." [matching chunk text]
```

**1c. Agent context** — Same enrichment for `format_search_results_for_llm()` in agent common.py.

### Files Modified
- `backend/app/core/search/hybrid.py` — `_vector_search()` returns chunk text
- `backend/app/core/chat/rag.py` — `_build_sources()` includes ratio/chunk text, `_format_context()` renders rich format
- `backend/app/core/agents/nodes/common.py` — `format_search_results_for_llm()` includes ratio + chunk text

## 2. Search Strategy Usage

### Problem
Query understanding outputs `search_strategy` but `hybrid_search()` ignores it.

### Solution

| Strategy | Behavior |
|----------|----------|
| `exact_match` | FTS only, skip vector search. For citation lookups. |
| `keyword_heavy` | RRF with FTS at 2x weight. For statute searches. |
| `balanced` | Current behavior unchanged. For general topics. |
| `vector_heavy` | RRF with vector at 2x weight. For conceptual searches. |

**Implementation:** Add `weights` parameter to `rrf_merge()` — list of floats parallel to `ranked_lists`. Each result's RRF contribution is multiplied by its list's weight. Default 1.0 (backward compatible).

For `exact_match`: try exact citation match in PostgreSQL `cases.citation` column first, fall back to FTS.

### Files Modified
- `backend/app/core/search/hybrid.py` — `rrf_merge()` weights, `hybrid_search()` strategy routing

## 3. Anti-Sycophancy + Prompt Hardening

### Problem
No prompt instructs the model to push back on incorrect legal assumptions. Stanford research shows legal AI agrees with users even when wrong.

### Solution

Add to all system prompts (chat, research, case prep):
- Anti-sycophancy: "If the user's legal premise appears incorrect based on retrieved precedents, flag this clearly before answering."
- Bench strength: "Include bench strength (single/division/constitution bench) when citing precedents."
- Disclaimer: "End substantive responses with: This is AI-assisted legal research, not legal advice."

Harden research prompts:
- Synthesis: "Rate each precedent as BINDING, PERSUASIVE, or DISTINGUISHABLE."
- Case prep: "Flag arguments that may be time-barred, jurisdictionally barred, or procedurally defective."

### Files Modified
- `backend/app/core/legal/prompts.py` — All prompt constants updated

## 4. Equivalent Citation Matching

### Problem
"AIR 1994 SC 1918" and "(1994) 3 SCC 1" are the same judgment — currently treated as unrelated.

### Solution

**New table:**
```sql
case_citation_equivalents
  id              UUID PK
  case_id         FK -> cases
  reporter        VARCHAR     -- SCC, AIR, SCR, INSC, CrLJ, SCALE, SCC OnLine
  citation_text   VARCHAR     -- "(1994) 3 SCC 1"
  year            INT
  UNIQUE(reporter, citation_text)
  INDEX on citation_text
```

**Ingestion:** Extract all citation formats from judgment text during ingestion, populate equivalence table.

**Search:** On `citation_lookup` intent, check equivalence table before hybrid search. Instant exact match.

**Display:** Show all known equivalent citations for a case.

### Files Created/Modified
- New: `backend/app/models/case_citation_equivalent.py`
- New: Alembic migration
- Modify: ingestion pipeline to populate equivalents
- Modify: `hybrid_search()` for citation lookup shortcut
- Modify: search API response to include equivalents

## 5. Structured Judgment Decomposition (iDRAF-style)

### Problem
Judgments stored as monolithic text. Can't search "within holdings only" or "within facts only."

### Solution

**New table:**
```sql
case_sections
  id              UUID PK
  case_id         FK -> cases
  section_type    VARCHAR     -- FACTS, ISSUES, ARGUMENTS, HOLDINGS, REASONING, ORDER
  content         TEXT
  section_index   INT
  summary         TEXT NULL   -- 1-2 sentence LLM summary
  INDEX on (case_id, section_type)
```

**Extraction:** Chunker already detects section boundaries. New: persist full section text to `case_sections`. Use LLM to extract HOLDINGS (from RATIO+ORDER) and REASONING (from ANALYSIS) as new synthesized sections with summaries.

**Section-aware search:** New `section` query param. For vector search: filter on Pinecone `section_type` metadata. For FTS: search `case_sections.content` instead of `searchable_text`.

**Section-aware RAG:** Prefer HOLDINGS and REASONING sections when building context.

### Files Created/Modified
- New: `backend/app/models/case_section.py`
- New: Alembic migration
- New: prompt in `prompts.py` for holdings/reasoning extraction
- Modify: ingestion pipeline
- Modify: `hybrid_search()`, `search_fulltext()` for section filter
- Modify: search API for section query param

## 6. Precedent Strength Classification + Confidence Scoring

### Problem
All precedents shown equally. A Constitution Bench decision and a single judge order look the same. Confidence scoring is naive (counts results).

### Solution

**Precedent strength** — Pure function, no LLM needed:

| Strength | Criteria |
|----------|----------|
| BINDING | Supreme Court, or same High Court with equal/larger bench |
| PERSUASIVE | Different High Court, larger bench of same court, tribunal |
| DISTINGUISHABLE | Factually distinct, different jurisdiction, obiter dicta |
| OVERRULED | Explicitly overruled by later decision |

Logic based on Indian precedent hierarchy:
- Constitution Bench > Full Bench > Division Bench > Single Judge
- Supreme Court binds all courts
- High Court binds only within its state

**Confidence scoring** — Replace naive heuristic:
```python
confidence = weighted_average(
    result_relevance=mean(reranker_scores[:5]),       # 40%
    coverage=cross_reference_ratio,                    # 20%
    source_authority=mean(precedent_strength_scores),  # 20%
    contradiction_penalty=1 - (contradictions/total),  # 20%
)
```

### Files Created/Modified
- New: `backend/app/core/legal/precedent_strength.py`
- Modify: `_format_context()`, agent common.py
- Modify: research agent `synthesize_memo_node` confidence formula
